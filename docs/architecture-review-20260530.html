After

RisParser

parse(InputStream) → List<BibliographicItem>

State machine only. Calls NormalizationCoordinator per field. No knowledge of file paths or normalization rules.

NormalizationCoordinator

normalize(fieldName, value, item)

Dispatches to per-domain normalization modules. Adding a new field type means adding one entry here — not modifying the parser.

RisWriter

write(items, mode, outputPath)

Write-only. Absorbs both mark-mode and remove-mode paths. One place to test output format.

Problem
IOService is shallow — its interface surface spans three unrelated concerns (parsing, Normalization coordination, writing), and its implementation pays no leverage dividend over calling the per-domain modules directly.
Solution
Extract RisParser (state machine), NormalizationCoordinator (dispatch), and RisWriter (output) as three deep modules each with a narrow interface.
Wins
  • • locality: RIS parsing bugs concentrate in one module
  • • leverage: test a single field type via parser interface
  • • delete duplicated write methods
  • • new field type = one-line change in coordinator
  • • interface shrinks per module

After

sequenceDiagram
  participant DS as DeduplicationService
  participant CP as ComparisonPhase
  participant ES as EnrichmentService
  DS->>CP: compare(items) → DuplicateSets
  Note over CP: pure: no state mutation
  DS->>ES: enrich(keptItem, duplicateSet)
  Note over ES: one seam, strategy per type
  Note over DS: label assignment separate
Problem
compareSet() simultaneously compares fields, manages Duplicate Set labels, and triggers Enrichment. Three separate concerns share one method with no seam between them — Enrichment logic for Reply and ClinicalTrials.gov Bibliographic Items is embedded mid-comparison.
Solution
Extract EnrichmentService with a narrow interface: enrich(kept, duplicates). The comparison phase returns Duplicate Sets; Enrichment runs in a separate pass over them.
Wins
  • • locality: Enrichment bugs (Reply, ClinicalTrials.gov) concentrate in one module
  • • leverage: test Enrichment independently of comparison
  • • comparison phase becomes testable as a pure function
  • • label assignment logic visible in one place
  • • Two-file mode reuses same Enrichment seam

Problem
Input data, transient deduplication state, and Enrichment output share one POJO. Deduplication state (label, isKeptBibliographicItem) leaks through the domain model's seam into modules that should not need it.
Solution
Make BibliographicItem immutable input-only. Introduce DeduplicationEntry as a wrapper holding mutable state and Enrichment output. Replace five boolean flags with a BibliographicItemType enum.
Wins

  • • leverage: comparison modules take immutable BibliographicItem — no mutation risk
  • • boolean explosion → type-safe enum; illegal combinations become unrepresentable
  • • interface of BibliographicItem shrinks; implementation absorbs nothing extra

Problem
ComparisonService is a shallow wrapper — its interface nearly matches its implementation (delegate to four injected modules). The deletion test: remove it, complexity reappears in DeduplicationService, not away. It has two static helpers (compareIssns, compareSameDois) that don't use the injected modules at all — mixed cohesion.
Solution
Either delete ComparisonService and have DeduplicationService hold the four per-field comparison modules directly, or wrap them in a record FieldComparators(…) with no extra logic — making the grouping explicit without adding a module.
Wins

  • • leverage: each comparison module now has one caller, not a relay
  • • mixed cohesion (instance delegates + static helpers) eliminated
  • • interface of the seam reduces to the four comparison modules directly

5. Internalize NormPatterns into per-domain normalization modules

Speculative in-process

Before — one monolithic registry

NormPatterns

50+ public static final Pattern

IOService
AuthorsNorm
TitlesNorm
JournalsNorm

All consumers share the same global namespace. No grouping by domain.

After — patterns private to each module

AuthorsNormalizationService private static final … (author patterns)
TitlesNormalizationService private static final … (title patterns)
JournalsNormalizationService private static final … (journal patterns)
NormalizationService private static final … (shared: accents, whitespace)

Each module owns its patterns. NormPatterns deleted.

Problem
NormPatterns is a grab-bag global registry — patterns for authors, journals, titles, pages, and DOIs share one class with no grouping. A maintainer changing an author pattern must grep to verify no other consumer uses it. The class has no interface: it's a namespace, not a module.
Solution
Move each pattern to private static final inside the normalization module that owns it. The small number of shared utility patterns (accent normalization, whitespace) stay in NormalizationService as private fields.
Wins
  • • locality: changing an author pattern touches one module
  • • unused patterns become visible as dead code at compile time
  • • NormPatterns class deleted
  • • caveat: patterns used by test code may need adjustment

Top recommendation

Start with #1 — Split the IOService fat module.

IOService is the deepest source of friction: three unrelated concerns share one shallow seam, its 369-line state machine can only be tested end-to-end, and two nearly-identical write methods drift independently. Splitting it into RisParser, NormalizationCoordinator, and RisWriter creates the leverage missing today: a single field's parsing is testable directly, the coordinator's dispatch table makes adding a new field type a one-line change, and the writer absorbs the duplicated mark/remove paths. Candidates #2 and #4 build naturally on top of it — once IOService is split, the Enrichment extraction and ComparisonService deletion become smaller, lower-risk changes.
